# Release Notes: v0.2.9

**Date**: 2025-11-10

Standards enhancement and schema expansion release incorporating real implementation feedback from forge-microtool-gimlet to refine forge standards, adding comprehensive cross-language Crucible documentation access patterns, and completing Wave 4 of experimental L'Orage Central DevSecOps schemas.

## Highlights

- **Forge Standards Enhancement**: API corrections and pattern clarifications based on forge-microtool-gimlet implementation
- **Enterprise Three-Layer Config Renaming**: Clearer positioning and explicit usage guidance

- **Microtool Repository Category**: New taxonomy for ultra-narrow, single-purpose CLI tools
- **Repository Naming Standard**: Comprehensive naming conventions for FulmenHQ repositories

## What's New

### 1. Forge Standards Enhancement Based on Real Implementation Feedback

forge-microtool-gimlet team provided three critical memos documenting API discrepancies and pattern ambiguities discovered during real implementation. All feedback incorporated.

#### Microtool Standard (`docs/architecture/fulmen-forge-microtool-standard.md`)

**API Corrections** (verified against gofulmen v0.1.10):

- **App Identity**: `appidentity.Get(ctx) ‚Üí (*appidentity.Identity, error)` with fields (not methods)
  - Correct: `identity.BinaryName`
  - Incorrect: `identity.BinaryName()` (compile error)
- **Logging**: `logging.NewCLI(serviceName)` for simple CLI tools
  - Use `zap.Field` helpers: `zap.String()`, `zap.Int()`, `zap.Error()`
  - Constants: `logging.TRACE`, `logging.DEBUG`, `logging.INFO`, `logging.WARN`, `logging.ERROR`
- **Signals**: `signals.OnShutdown()`, `signals.EnableDoubleTap()`, `signals.Listen()`
- **Exit Codes**: `foundry.ExitSuccess`, `foundry.ExitFailure`, `foundry.ExitConfigInvalid`

**Configuration Pattern Overhaul**:

- **Removed**: Three-Layer Config (RECOMMENDED)
- **Added**: Simple Config Pattern (REQUIRED)
- **Rationale**: Microtools NEVER use Enterprise Three-Layer Config
- **Pattern**: Two-layer only: Defaults ‚Üí User Overrides (env vars, flags, optional config file)
- **Graduation Criteria**: If microtool needs enterprise config ‚Üí promote to `cli` category

**Crucible Shim Refinement**:

- **Changed**: RECOMMENDED ‚Üí CONDITIONAL
- **REQUIRED for**: Schema validators, taxonomy tools, commit hooks, doc generators, CI validators
- **Skip for**: Tools without Crucible asset dependencies (most microtools like gimlet)
- **Rationale**: Provides offline access to meta-schemas (JSON Schema Draft 07, 2020-12) and SSOT data

**Reference Implementation Section**:

- Added section pointing to forge-microtool-gimlet repository
- Key files: `cmd/gimlet/main.go`, `internal/cmd/root.go`, `internal/runtime/logging.go`, `internal/config/loader.go`
- Validated against gofulmen v0.1.10 and real build/test/run workflows

**Documentation Access Examples**:

- Commit validation hooks accessing agentic-attribution.md
- Doc generators accessing coding standards
- CI compliance checkers accessing repository structure
- Template refit automation accessing CDRL workflow

#### CDRL Template Standard (`docs/architecture/fulmen-template-cdrl-standard.md`)

**Working Implementation Pattern** (new comprehensive section):

- **Build Requirements**: Templates MUST compile/build/test/run as-is
- **IDE Experience**: Syntax highlighting, type checking, auto-complete MUST work
- **CI/CD Requirements**: Templates run linting, testing, building in CI
- **Comparison Table**: Fulmen Forge vs Literal Templates (Cookiecutter/Yeoman)

**Anti-Pattern Documentation**: NO `.template` file suffixes

- Templates ship working files (`go.mod`, `.fulmen/app.yaml`), NOT `.template` variants
- Users edit in place during CDRL refit
- Validation catches hardcoded references, no manual find-replace
- **Why**: Templates must be buildable and testable in CI; `.template` files break builds and IDEs

**Clarification**: Parameterization points are NOT string literals

- Edit `.fulmen/app.yaml` first (single source of truth)
- Run `make validate-app-identity` (finds hardcoded breed name)
- Update identified locations (module path, imports, directory names)

#### Workhorse/Codex Standards

- Updated Crucible Shim sections with documentation access APIs
- Added use case examples:
  - **Workhorse**: Runtime compliance validation, API documentation generation, operational playbooks
  - **Codex**: Searchable content ingestion, API reference generation, schema registry displays

### 2. Enterprise Three-Layer Config Module Renaming

**Renamed**: `three-layer-config.md` ‚Üí `enterprise-three-layer-config.md`

**Purpose**: Clearer positioning and explicit guidance on when to use/not use

**Module Spec Updates** (`docs/standards/library/modules/enterprise-three-layer-config.md`):

- **When to use**:
  - ‚úÖ Workhorse services (HTTP APIs, background workers)
  - ‚úÖ Complex CLI tools with multiple configuration sources
  - ‚úÖ Applications requiring Crucible SSOT defaults for compliance

- **When NOT to use**:
  - ‚ùå Microtools (use Simple Config Pattern instead)
  - ‚ùå Simple applications with minimal configuration
  - ‚ùå Prototypes and experiments

**Three Layers Clarification**:

1. **Layer 1: SSOT Defaults** - Crucible assets (ecosystem-wide standards)
2. **Layer 2: User Config** - Application-specific settings (`~/.config/{vendor}/{app}/config.yaml`)
3. **Layer 3: Runtime Overrides** - Environment variables and CLI flags

**Precedence**: Layer 3 (runtime) > Layer 2 (user) > Layer 1 (SSOT)

**Cross-References Updated**: 13 files across all forge standards, helper library docs, module manifests

### 5. Microtool Repository Category

New taxonomy category for ultra-narrow, single-purpose CLI deployment/automation tools.

#### Category Definition (`config/taxonomy/repository-categories.yaml`)

**Summary**: Hyper-focused, single-purpose CLI tools for deployment automation

**Strict Architectural Constraints**:

- **MUST** have single primary purpose (e.g., fixture deployment, config synchronization)
- **MUST** be written in Go, TypeScript/Bun, or Rust only
- **SHOULD** import helper library (gofulmen, tsfulmen, or rsfulmen)
- **MUST NOT** export packages or be imported by other repositories (one-way dependency flow)
- **MUST** obey exit codes, signals, logger, config standards

**Examples**: grinder (Go), tongs (TypeScript), rasp (Rust)

**Language-Matching Naming Pattern**:

Template naming follows `forge-microtool-{instrument}` where instrument first letter matches language:

- **G-instruments for Go**: grinder, gauge, gouge
- **T-instruments for TypeScript**: tongs, tap, trammel
- **R-instruments for Rust**: rasp, reamer, router

**Schema Updates**:

- Added `microtool` to category key enum (`schemas/taxonomy/repository-category/v1.0.0/category-key.schema.json`)

**Validation Support**:

- Updated `scripts/validate-schemas.ts` to recognize microtool category

**Microtool Standard** (`docs/architecture/fulmen-forge-microtool-standard.md`):

Comprehensive standard covering:

- Language constraints (Go, TypeScript/Bun, Rust only)
- Helper library requirements and module compliance
- Directory structure and file organization
- Required modules (App Identity, Simple Config, Logging, Exit Codes, Signals, Error Handling, Pathfinder)
- Security considerations (no privilege escalation, safe filesystem operations)

## Migration

### Update Dependency

**Go**:

```bash
go get github.com/fulmenhq/crucible@v0.2.9
go mod tidy
```

**TypeScript/Bun**:

```bash
bun add @fulmenhq/crucible@latest
```

**Python**:

```bash
uv add crucible@latest
```

### Code Changes (Microtools Only)

If you're developing a microtool and were following the v0.2.8 standard:

**1. Update API Calls** (if using incorrect patterns):

```go
// OLD (incorrect - methods)
name := identity.BinaryName()

// NEW (correct - fields)
name := identity.BinaryName
```

**2. Switch to Simple Config Pattern** (if using three-layer):

```go
// OLD (three-layer config - too complex for microtools)
cfg := config.NewThreeLayer(crucible.GetDefaults(), userConfig, envOverrides)

// NEW (simple config - two layers only)
cfg := &Config{InputPath: "."}  // defaults
if file.Exists() { load(file, cfg) }  // user config
if env := os.Getenv("INPUT"); env != "" { cfg.InputPath = env }  // env overrides
```

**3. Update Crucible Shim Usage** (if applicable):

Only use if accessing SSOT assets (schemas, docs, taxonomies). Skip if your microtool has no Crucible dependencies.

### Documentation Access (New Capability)

All languages can now access Crucible documentation programmatically:

**Go**:

```go
import "github.com/fulmenhq/gofulmen/crucible"

doc, err := crucible.GetDoc("standards/agentic-attribution.md")
```

**Python**:

```python
from pyfulmen import crucible

doc = crucible.get_doc("standards/agentic-attribution.md")
```

**TypeScript**:

```typescript
import { crucible } from "@fulmenhq/tsfulmen";

const doc = await crucible.getDoc("standards/agentic-attribution.md");
```
