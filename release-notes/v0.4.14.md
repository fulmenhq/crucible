# Release v0.4.14

**Adds a `metadata.typescript` packaging section to app-identity, finishes the Python enum `StrEnum` modernization at the generator, and moves CI off the Node 20 runtime.**

## Overview

A small, additive release. The headline is a new optional `metadata.typescript` section in the app-identity schema so TypeScript/npm packages can declare their npm package name and executable bin entries first-class — the npm analogue of the existing `metadata.python`. It also closes the loop on v0.4.13's enum modernization (which updated the generated files but not the codegen templates), and brings the four CI workflows onto the Node 24 action runtime ahead of GitHub's force-migration.

No breaking changes; the app-identity schema stays `v1.0.0`.

## Highlights

### app-identity: `metadata.typescript` packaging section (Added)

- New optional `metadata.typescript` object on the app-identity `v1.0.0` schema, mirroring `metadata.python`:
  - `package_name` — npm package name (scoped or unscoped, e.g. `@fulmenhq/tsfulmen`)
  - `console_scripts: [{ name, entry_point }]` — executable bin entries that map to package.json `bin` (`entry_point` is the path to the built executable, not a `module:function`)
- **Additive and backward-compatible**: `metadata.additionalProperties` was already `true`, so no existing document is invalidated and the schema version stays `v1.0.0`. Metadata parsing is pass-through across Go/Python/TypeScript, so no parser changes are required (confirmed: `custom_field` survives in the parity snapshot).
- Section named `typescript` for consistency with the language-named convention (`metadata.python` ↔ `lang/python`; this ↔ `lang/typescript`).
- Includes: standard-doc metadata-table row + a "With TypeScript/npm Packaging" example + a corrected vendor-vs-package-name pointer; example YAML block; a new `typescript-package` valid fixture; and a `typescript_package` parity-snapshot case. Propagated to all language wrappers.
- Motivated by tsfulmen v0.3.1, which ships three `bin` commands and needed a first-class home for them (replacing an interim custom `metadata.console_scripts` field).

### codegen: Python enum templates emit `StrEnum` (Fixed)

- v0.4.13 modernized the generated `fulpack`/`fulencode` Python enums `(str, Enum) → StrEnum` (ruff `UP042`, Python floor 3.12) but left the **codegen templates** emitting `class X(str, Enum)`. As a result, every `make sync`/`make precommit` regenerated and reverted them, leaving local precommit effectively unrunnable.
- Both templates now emit `from enum import StrEnum` + `class X(StrEnum)`, matching the existing fulhash template. Regeneration is now idempotent and ruff-clean.

### CI: GitHub Actions off the Node 20 runtime (Changed)

- `actions/checkout` v4→v5, `actions/setup-go` v5→v6, `oven-sh/setup-bun` v1→v2, `astral-sh/setup-uv` v3→v7 — each verified node24 via its `action.yml`, ahead of GitHub's node20 force-migration.
- `Swatinem/rust-cache@v2` is already node24 (kept). `dtolnay/rust-toolchain` is a composite action (no node runtime); pinned `@master → @v1` (a stable version tag that still honors `toolchain: "1.89"`) for supply-chain hygiene.
- **goneat pin** `v0.5.12 → v0.5.13` (Makefile), matching the current release and local toolchain.

## Breaking Changes

**None.** The app-identity schema addition is additive (optional field, `v1.0.0` unchanged); the codegen and CI changes alter no published asset content or public API.
