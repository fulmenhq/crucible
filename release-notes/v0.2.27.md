### Problem

The `vendor` field pattern in `app-identity.schema.json` previously required an alphabetic first character (`^[a-z][a-z0-9]{0,62}[a-z0-9]$`), which prevented valid vendor names like `3leaps`, `37signals`, and `8x8`.

### Solution

Updated the vendor pattern to `^[a-z0-9]{2,64}$`, permitting leading digits.

### Cross-Language Safety Analysis

This change is safe because the `vendor` field is used for:

- **Filesystem paths** (`~/.config/<vendor>/`) - All filesystems accept leading digits
- **Configuration directories** - No language restrictions apply
- **Namespace grouping** - Organizational identifier, not a code symbol

The `vendor` field is NOT used as a language-specific package identifier:

| Language   | Concern                              | Impact | Notes                          |
| ---------- | ------------------------------------ | ------ | ------------------------------ |
| Go         | Module paths can start with digits   | Safe   | `github.com/3leaps/pkg` works  |
| Python     | Identifiers cannot start with digits | Safe   | Vendor not used as import name |
| TypeScript | npm scopes can start with digits     | Safe   | `@3leaps/pkg` is valid         |
| Rust       | Crate names can start with digits    | Safe   | Cargo accepts numeric prefixes |

Python projects requiring identifier-safe names use `metadata.python.package_name`, which retains its alphabetic-first requirement.
