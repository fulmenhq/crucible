# Release v0.4.15

**Completes the ADR-0012 absolute-`$id` cross-reference rollout for the logging and module-manifest schemas, migrates the logging `$id`s to the canonical version-in-path form, and refreshes the vendored 3leaps/crucible pin to v0.1.14.**

## Overview

A schema-correctness and hygiene release. The headline fixes a class of cross-schema `$ref` resolution failures in `observability/logging` and `library/module-manifest` that broke memory-based validators (reproduced from tsfulmen and gofulmen) — finishing the partial [ADR-0012](../docs/architecture/decisions/ADR-0012-schema-ref-ids.md) rollout that had only converted `logger-config`. In the same pass the logging schema `$id`s move from the deprecated version-in-filename form to the canonical version-in-path form, so on-disk layout matches identity per the [Canonical URI Resolution Standard](../docs/standards/publishing/canonical-uri-resolution.md). Rounding out the release: a vendored upstream refresh to the just-released 3leaps/crucible v0.1.14, and a small release-tooling cleanup.

No fulmenhq public API changes and no schema version bumps; all touched schemas remain `v1.0.0`.

## Highlights

### schemas: complete the ADR-0012 absolute `$id` cross-references (Fixed)

- ADR-0012 (use absolute `$id` URLs for cross-schema `$ref`) was only partially implemented — an earlier commit converted `logger-config.schema.json` (the case gofulmen hit) but left relative cross-file `$ref`s in `log-event`, `severity-filter`, and `middleware-config`, plus an off-by-one relative `$ref` in `library/module-manifest`. Relative cross-file refs do not resolve in memory-based validators (no filesystem context), which is the exact failure ADR-0012 exists to prevent.
- All 16 remaining relative cross-file `$ref`s across those 4 schemas (12 in `log-event`, 2 in `severity-filter`, 1 in `middleware-config`, 1 in `module-manifest`) are now absolute canonical `$id` URLs; `logger-config`'s existing absolute middleware ref was repointed to the new version-in-path `$id`. The `module-manifest` → `taxonomy/language` reference (`../../taxonomy/...`, which resolved one level short) is fixed and converted to the absolute form. Internal `#/$defs/...` refs remain relative (ADR-0012 permits).

### observability/logging: `$id`s migrated to canonical version-in-path (Changed)

- The 6 logging `$id`s moved from the deprecated version-in-filename form (`.../observability/logging/log-event-v1.0.0.json`) to canonical version-in-path (`.../observability/logging/v1.0.0/log-event.schema.json`), matching the on-disk layout and the rest of the corpus.
- This is an identity change **inside** the existing `v1.0.0` files (the schema contents and version are unchanged). See Breaking Changes for the legacy-URI compatibility note.

### upstream: vendored 3leaps/crucible v0.1.12 → v0.1.14 (Changed)

- Refreshed the vendored upstream to the just-released 3leaps/crucible **v0.1.14** (commit `18018788`) so the fulmenhq SSOT — and downstream consumers that track it — line up with a released upstream.
- Content delta within the synced scope is minimal: the vendored schemas (`classifiers`/`foundation`/`ailink`/`agentic` `v0`) are byte-identical; only 3 classifier docs received one-line wording reconciliations, plus the regenerated `PROVENANCE.md`. Upstream's new `auth/v0` schema and `dataeng` role fall outside the sync scope and are not vendored here.

### tooling: model-agnostic upstream-pull provenance (Changed)

- `scripts/3leaps-crucible-upstream-pull.ts` no longer hardcodes a specific model version in the generated `PROVENANCE.md` "Synced By" line (it went stale across model changes); the precise model is recorded in the commit attribution trailer.

## Breaking Changes

**None to public API or schema versions** — all touched schemas remain `v1.0.0` and additions are backward-compatible at the contract level.

**Compatibility note (logging `$id` identity):** the `observability/logging` schemas' canonical `$id` URIs changed from the version-in-filename form to version-in-path _within_ `v1.0.0`. The legacy URIs (`.../logging/<name>-v1.0.0.json`) are no longer emitted. No in-repo consumers referenced the old URIs (verified), and the change is what makes cross-file refs resolvable. Any external consumer that resolved these schemas by their old canonical URI should switch to the new version-in-path URIs; spechost deployments should maintain legacy→canonical redirects during the transition if applicable.
